# Principles Enforcement Roadmap

Automate enforcement of the architectural principles documented in `docs/principles.md`. Today, conformance is checked only by human review and by a handful of existing lint rules (ruff `PLC0415` for top-level imports). This roadmap adds three layers of enforcement — pytest architecture tests for structural invariants, ruff rules for syntactic patterns, and an LLM adversarial review agent for semantic/intent-level checks — so that violations are caught automatically rather than discovered during code review.

## Phase 1: Layer dependency tests

Add pytest-based architecture tests that verify the unidirectional layer dependency rule (principle 8) and the module decoupling rule (principle 1).

### Context

The codebase has a clean layered architecture: CLI/Agent → Services → Repos → Domain. Today this is enforced only by convention. A single careless import — say, a service importing from `cli/` — would violate the architecture silently. Import-graph tests make this a hard failure.

### Steps

1. Create `tests/architecture/` directory with a `test_layer_dependencies.py` module.
2. Write a helper that, given a package path (e.g., `fantasy_baseball_manager.services`), collects all imports from all `.py` files in that package using the `ast` module.
3. Define the allowed dependency rules:
   - `domain/` → standard library, other `domain` modules only.
   - `repos/` → `domain`, `repos.errors`, standard library. Never `services`, `cli`, `agent`, `tools`.
   - `services/` → `domain`, `repos.protocols`, `models.protocols`, `features.protocols`, standard library. Never `cli`, `agent`, `tools`.
   - `ingest/` → `domain`, `repos.protocols`, standard library, third-party HTTP libs. Never `cli`, `agent`, `services`.
   - `models/<subpackage>/` → `domain`, `models.protocols`, `models.registry`, `features`, standard library. Never `cli`, `agent`, `repos` concrete classes.
   - `cli/`, `agent/`, `tools/` → anything (composition roots).
4. Write one parametrized test per layer that asserts no imports from forbidden packages.
5. Add a test that specifically verifies services import repo *protocols*, not concrete `Sqlite*` classes (principle 1).
6. Verify all tests pass on the current codebase (these should be green from day one — they codify the existing state).

### Acceptance criteria

- `tests/architecture/test_layer_dependencies.py` exists and passes.
- A simulated violation (e.g., a service importing from `cli`) would cause a test failure (verified with a negative test or documented manually).
- Tests use `ast` to parse imports — no runtime import side effects.
- Tests run in under 2 seconds (they scan files, not execute code).

## Phase 2: Domain purity tests

Add tests enforcing that domain models are frozen dataclasses with no behavior and no infrastructure imports (principle 7).

### Context

Domain models are the innermost layer. They must remain pure data — frozen, no methods, no I/O. A domain model that grows a `save()` method or imports `sqlite3` would violate multiple principles. These tests make that structurally impossible.

### Steps

1. Create `tests/architecture/test_domain_purity.py`.
2. Write a test that scans all `.py` files in `domain/` and collects every class definition using `ast`.
3. Assert every class decorated with `@dataclass` uses `frozen=True`.
4. Assert no class defines methods other than those generated by `dataclass` (allow `__post_init__`, `__str__`, `__repr__` if needed, but flag anything else — especially methods that take `self` and do I/O).
5. Assert imports in `domain/` modules are restricted to: `dataclasses`, `typing`, `enum`, `statistics`, `collections`, `decimal`, `math`, `datetime`, `re`, other `domain` modules, and `__future__`. No `sqlite3`, no `httpx`, no project infrastructure packages.
6. Verify green on the current codebase.

### Acceptance criteria

- `tests/architecture/test_domain_purity.py` exists and passes.
- Every domain dataclass is verified frozen.
- Custom methods on domain classes are flagged (or an explicit allowlist is maintained).
- Domain import restrictions are enforced.

## Phase 3: Ruff rule expansion

Expand the ruff configuration to catch additional principle violations at lint time, before tests even run.

### Context

The current ruff config selects only `E`, `F`, and `PLC0415`. Several additional ruff rules align directly with the principles document and can be enabled with zero or minimal codebase changes.

### Steps

1. Evaluate and enable these rule sets:
   - `S` (flake8-bandit) — catches hardcoded passwords, `exec()`, and other security issues (principle 3: secrets via env vars). May need targeted `noqa` for test fixtures.
   - `T20` (flake8-print) — catches stray `print()` calls that should be logging or CLI output.
   - `UP` (pyupgrade) — enforces modern Python syntax (project requires 3.14+).
   - `B` (flake8-bugbear) — catches common bugs and design problems.
   - `SIM` (flake8-simplify) — catches unnecessarily complex code patterns.
   - `I` (isort) — enforces consistent import ordering.
   - `TCH` (flake8-type-checking) — moves type-only imports behind `TYPE_CHECKING` blocks, reducing runtime import weight.
2. For each rule set, run `uv run ruff check --select <RULE> src tests` to measure existing violations.
3. Fix violations or add targeted `noqa` comments where the rule conflicts with project patterns.
4. Add the rule set to `pyproject.toml` `[tool.ruff.lint] select`.
5. Evaluate `INP` (flake8-no-pep420 implicit namespace packages) to ensure all packages have `__init__.py`.
6. Add a `[tool.ruff.lint.per-file-ignores]` section for test-specific allowances (e.g., `S101` for assert in tests is already fine since pytest uses assert).

### Acceptance criteria

- At least `S`, `B`, `UP`, and `I` rule sets are enabled in `pyproject.toml`.
- `uv run ruff check src tests` passes with the expanded rule set.
- No new `noqa` comments are added without a justifying comment.
- Pre-commit hook continues to pass.

## Phase 4: Module re-export audit tests

Add tests that verify each package exposes a public API through `__init__.py` and that external consumers use the package-level import (principle 6).

### Context

Most `__init__.py` files are currently empty. Principle 6 is aspirational — it says consumers should import from the package root. This phase first populates the re-exports, then adds a test to prevent regression.

### Steps

1. For each major package (`domain`, `repos`, `services`, `models`, `ingest`, `features`), audit which symbols are imported by other packages.
2. Add re-exports to each package's `__init__.py` for all public symbols. Use explicit `from .submodule import ClassName` and define `__all__`.
3. Update consumers to import from the package root instead of reaching into submodules. (This is the largest step — may need to be done package by package.)
4. Create `tests/architecture/test_reexports.py` that for each major package:
   - Verifies `__all__` is defined.
   - Verifies every symbol in `__all__` is importable from the package root.
   - Verifies no internal-only symbols leak into `__all__`.
5. Optionally, add a test that scans all `from fantasy_baseball_manager.<pkg>.<submodule> import X` in non-composition-root code and flags imports that bypass the package root when a re-export exists.

### Acceptance criteria

- Each of the 6 major packages has a populated `__init__.py` with `__all__`.
- Existing tests and CLI continue to work after import path changes.
- `tests/architecture/test_reexports.py` verifies re-export completeness.
- A regression test ensures new public symbols are added to `__all__`.

## Phase 5: LLM adversarial review agent

Build a lightweight review agent that checks diffs against the principles document for violations that static analysis cannot catch — design intent, naming quality, separation of concerns, model/mode generality, and interface simplicity.

### Context

Principles 4 (model generality), 5 (interaction mode generality), and 11 (simple functional interfaces) are inherently semantic. No lint rule can detect "this CLI command hardcodes a model name" or "this service contains business logic that should be in a shared module." An LLM agent can read the principles and the diff, then flag violations with references to specific principles.

### Steps

1. Create `scripts/arch_review.py` that:
   - Accepts a diff on stdin or from a file path argument (default: `git diff HEAD~1`).
   - Loads `docs/principles.md` as the architectural spec.
   - Sends the diff + principles to Claude with a system prompt instructing it to act as an adversarial architecture reviewer.
   - Outputs violations as structured findings: principle number, file, line range, description, severity (error/warning).
   - Exits non-zero if any error-severity violations are found.
2. Write the system prompt to be specific:
   - Check for upward layer imports (redundant with tests, but catches naming tricks).
   - Check for model-specific or mode-specific code paths.
   - Check for business logic in interaction-layer code.
   - Check for complex protocols that could be simplified.
   - Check for domain models with behavior.
   - Check for hardcoded secrets or credentials.
3. Add a Claude Code hook (`.claude/hooks.json`) that runs the review on post-commit, providing immediate feedback during development.
4. Add a GitHub Actions workflow step that runs the review on PRs and posts findings as PR comments using `gh pr comment`.
5. Write tests for the script's diff-parsing and output-formatting logic (not for the LLM judgments themselves — those are tested by running the agent on known-good and known-bad diffs and spot-checking).

### Acceptance criteria

- `scripts/arch_review.py` runs against a diff and produces structured findings.
- Known violations (manually crafted test diffs) produce expected findings.
- Clean diffs produce no findings.
- Claude Code hook runs the review automatically on commit.
- GitHub Actions workflow posts findings on PRs.
- The agent references specific principle numbers in its output.

## Ordering

**Phase 1 → 2** can be done in either order — both are independent pytest architecture tests. Phase 1 is recommended first since layer violations are the highest-risk architectural defect.

**Phase 3** is independent of phases 1-2 and can be done in parallel. It extends the existing lint infrastructure rather than adding new test files.

**Phase 4** depends loosely on phases 1-2 being done first. The import-path migration (step 3) is a large refactor, and having the layer dependency tests in place first ensures the migration doesn't accidentally introduce violations.

**Phase 5** depends on all prior phases being complete. The LLM agent is most valuable when the static checks already handle the easy cases — it focuses on the semantic violations that escape linting and architecture tests. It also benefits from the principles being battle-tested through phases 1-4.

```
Phase 1 (layer deps) ──┐
                        ├──► Phase 4 (re-exports) ──► Phase 5 (LLM agent)
Phase 2 (domain) ──────┘
Phase 3 (ruff) ─────────────────────────────────────►
```
